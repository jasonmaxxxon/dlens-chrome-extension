# DLens 0.2 — ViewModel Boundary plan

> Date: 2026-06-11 · Author: Claude (plan) for Codex (execution) · Baseline: `main` @ `4ba1e77` (0.1.33, 701/701)
> Predecessor: `docs/handoff/2026-06-11-invariant-consolidation-plan.md` (slices ①–⑨ — the single-truth classifiers this phase composes)

## The one cut

UI must stop reading raw `SessionItem` / `Signal` / backend capture and re-deriving "is this ready / stale / fallback / analyzable". Instead:

> **UI boundary = { ViewModel in, Command out }. Nothing else crosses it.**
> - **ViewModel (down):** a pure, composed, render-ready object. UI renders it; it does not derive it.
> - **Command (up):** a typed action carrying an **explicit target** (builds on slice ⑤). UI dispatches it; it does not mutate storage.

After this, backend shape changes / storage migration / capture-shape changes never force the UI to re-guess. The UI's only jobs: **present state, dispatch command.**

## Responsibility boundaries (the 0.2 contract)

| Layer | Owns | Must NOT |
|-------|------|----------|
| **Backend** (`dlens-ingest-core`, private) | crawl, ThreadReadModel, OP/reply split, cluster/raw evidence, job status | — |
| **Extension state** (`src/state`) | storage, session/signal/topic identity, readiness, staleness, provenance, mutation transaction | leak raw shapes upward |
| **Analysis** (`src/compare`) | Product/Topic/Compare/PR prompt, schema, fallback, cache | — |
| **UI** (`src/ui`) | render a ViewModel, dispatch a Command | read raw `SessionItem`/`Signal`/capture; derive readiness/stale/fallback; mutate storage |

## Why this is lower-risk than it sounds: the VM is composition, not new truth

Slices ①–⑨ already built the single-truth classifiers. The ViewModel is the **plate**; those are the **ingredients**. This phase is mostly wiring + deleting the UI's ad-hoc derivation.

| `ProductSignalViewModel` field | Source classifier (already on main) |
|---|---|
| `sourcePreview` | `projectCapturedPost()` — `src/state/captured-post.ts` |
| `readiness` | `buildSignalReadinessById()` — `src/state/signal-readiness.ts` |
| `analysisState` (`missing/queued/ready/stale/failed`) | `src/state/load-state.ts` + `src/state/derived-record.ts` (stale) |
| `provenance` (`ai/fallback/missing`) | `src/state/ai-provenance.ts` |
| `actions` | typed Commands w/ explicit target (slice ⑤ seam, `src/state/session-signal-seam.ts`) |

The extension-side **read-model seam already exists** (`captured-post.ts`) — that's why the backend half is a *separate* track (below), not part of this phase.

## Contracts to define first (before touching any UI)

New layer: **`src/viewmodel/`** (pure, no `chrome.*`, no `fetch`, unit-testable).

```ts
// src/viewmodel/product-signal.ts
export type AnalysisState = "missing" | "queued" | "ready" | "stale" | "failed";
export type Provenance = "ai" | "fallback" | "missing";

export type ProductSignalViewModel = {
  signalId: string;
  title: string;
  sourcePreview: EvidencePreview;     // from projectCapturedPost
  readiness: SignalReadiness;          // from signal-readiness
  analysisState: AnalysisState;        // from load-state + derived-record
  provenance: Provenance;              // from ai-provenance
  actions: ProductSignalAction[];      // typed command descriptors (see below)
};

// Command (up): a descriptor, NOT a closure over raw state.
export type ProductSignalAction =
  | { kind: "analyze"; target: { sessionId: string; signalId: string } }
  | { kind: "recrawl"; target: { sessionId: string; signalId: string } }
  | { kind: "generateReading"; target: { sessionId: string; signalId: string } }
  | { kind: "remove"; target: { sessionId: string; signalId: string } };

// The pure builder — the whole point.
export function buildProductSignalViewModels(snapshot: ExtensionSnapshot): ProductSignalViewModel[];
```

Rules for the contract:
- `actions` are **command descriptors** (kind + explicit target), not `() => void` callbacks. The shell maps `kind` → `sendAndSync(message)`. This keeps the VM pure and serializable, and the target explicit (slice ⑤).
- The VM is the **only** thing a workspace component receives for data. No more `signals`, `analyses`, `items`, `readinessById` props threaded in.

## Migration recipe (per workspace — this is how each slice runs)

1. **Build the VM beside the UI.** Write `buildXxxViewModels(snapshot)` as a pure fn composing existing classifiers. No UI change yet.
2. **Characterization-test it == current UI derivation.** Assert the VM reproduces exactly what the current component derives today (counts, readiness labels, stale/fallback flags) on real-shaped fixtures (incl. uncrawled / legacy / missing-field). Red-green before swapping.
3. **Swap the UI to read the VM**, and in the same PR **delete** the component's ad-hoc derivation (the `safe* = Array.isArray(...)`, inline readiness reads, count loops, stale guesses). Collapse-don't-add — net LOC should drop.
4. **Route `actions` through the shell** as typed commands. UI dispatches `kind`, never builds a message inline.

## Sequence (Claude's recommendation)

**Product first, as the reference template** — `ProductSignalView` exercises every state (readiness × analysisState × provenance) and is the most-touched surface, so it proves the pattern before replication.

```
1. src/viewmodel/product-signal.ts  + characterization tests   ← define contract
2. ProductSignalView swap to VM + delete ad-hoc derivation     ← reference migration
3. TopicViewModel        (TopicDetailView)        ← replicate
4. CompareViewModel      (CompareView)            ← replicate
5. PrEvidenceViewModel   (PrEvidenceViews)        ← replicate
```

Do **not** do all four workspaces at once. One VM per PR, each green, each deletes its UI's derivation.

**Parallel track (different owner, different repo) — Backend ReadModel:**
- The extension already has its read-model seam (`captured-post.ts`). What remains is backend-side: (a) formalize the wire schema between `dlens-ingest-core` and the extension; (b) **fix OP-continuation / discussion-reply splitting** (no duplicate root content, OP chatter not mislabeled as continuation).
- (b) is the **product-value lever** (analysis trustworthiness) — higher user-visible ROI than any UI work. Run it in parallel; do not block the ViewModel phase on it, and do not bundle it in.

## Governing rules (carried from the consolidation plan — still apply)

1. One workspace VM per slice/PR. Never bundle two workspaces.
2. Characterization-test-first; add invariant tests (`actions implies valid target`, `analysisState=stale implies a reading exists`).
3. Collapse, don't add — each slice deletes the UI derivation it replaces (grep clean).
4. VM lives in `src/viewmodel/`, pure, no `chrome.*`/`fetch`, unit-testable.
5. Branch off `main`, one PR each, green (`typecheck` + full suite + `build`). **No version bump inside a slice** — bumps are separate release commits.
6. Do NOT touch: storage backward-compat normalize (behind migration gate), backend polling/backoff, error boundaries.
7. **Visual revamp waits** until the VM boundary is in place — otherwise the old model's implicit coupling gets carried into the new UI. (User decision, 2026-06-11.)

## Definition of done (per workspace slice)

- `src/viewmodel/<workspace>.ts` pure builder + characterization tests (VM == prior UI derivation) + ≥1 invariant test.
- Component receives **only** the VM (+ a command dispatcher); its ad-hoc derivation is deleted (grep clean).
- `actions` flow as typed commands with explicit targets; UI builds no inline messages.
- `typecheck` + full suite + `build` green; branch off `main`, one PR; no version bump.
- One line appended here: slice ✓, what was deleted, which classifiers the VM composes.

## Open frontiers (intentionally deferred — not this phase)

- **Hover DOM extraction** (`findCardCandidate` / `buildTargetDescriptor`, content script): a different class — an inherently heuristic extractor, not scattered guesses of one truth. Harden with **live Threads-DOM fixture replay**, not a classifier. Do only if hover bugs recur.
- **Full storage transaction (⑥ complete):** highest blast radius, deliberately partial (`session-signal-seam.ts` covers signal-delete cross-key). Deepen only when the spine/VM surfaces a real consistency need, behind a migration gate.

## Pipeline spine (next-next, after VM boundary proves out)

Once UI reads VMs and dispatches Commands, the typed `hover.detected → preview.confirmed → signal.saved → crawl.queued → capture.ready → analysis.ready → ui.ready` spine + trace harness (the formalized `qa-trace`) becomes the natural follow-on: every step a typed event/result, debug by reading the trace not guessing the UI. Sequenced after the Product VM proves the boundary.
