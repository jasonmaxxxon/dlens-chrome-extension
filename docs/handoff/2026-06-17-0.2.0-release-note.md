# DLens 0.2.0 release note

Date: 2026-06-17
Baseline: extension `0.1.33` at `8478de0` · backend `dlens-ingest-core`
`main` at `6d0cb70`
Head: extension `0.2.0` at `6bd17f3` · backend `main` at `f816551`

## Why 0.2.0

0.1 → 0.2 is a minor bump. No breaking change to existing extension
storage shape, no breaking change to backend contracts. All shipped
work is additive: new endpoints, new typed contract fields, new UI
surfaces, new test/fixture guards.

Two reasons it earns a minor bump rather than another patch:

- The work-truth contract between extension and backend changed shape
  (additive but visible): `/worker/status` now carries 12 new fields
  and a new `BackendWorkUiState` projection drives view recovery
  copy; `/health` is a new endpoint with its own latency contract.
- A new visible UI surface (backend health dot in
  `ProcessingStrip`) was added, plus an extension descriptor field
  (`engagement.source`) that downstream surfaces can read.

`0.2.0` (not bare `0.2`) is required because Chrome MV3
`manifest.version` rejects single-segment values. Version is locked
across `package.json`, `package-lock.json`, `wxt.config.ts:20`,
`src/ui/version.ts:BUILD_VERSION`, `tests/manifest-config.test.ts`,
and the build output at `output/chrome-mv3/manifest.json`.

## What shipped — extension (`dlens-product-latest`)

### Work-truth projection + recovery copy (API/JOBS PRs)

- `WorkerStatusResponse` in `src/contracts/ingest.ts` gained optional
  `pending_due_jobs / retry_scheduled_jobs / running_jobs /
  expired_running_jobs / dead_jobs / pending_analyses /
  running_analyses / failed_analyses / earliest_retry_at /
  next_due_at / last_drain_error / last_drain_finished_at`.
- `projectBackendWorkStatus()` in `src/state/processing-state.ts`
  collapses the new fields into a `BackendWorkUiState` union with
  explicit priority `backend_error > expired_running > analysis_failed >
  retry_waiting > analysis_waiting > draining > idle`. Views consume
  the union, never the raw counts.
- `reconcileSessionItem()` in `src/state/store-helpers.ts` now
  promotes a failed analysis snapshot into the canonical item error
  path (`lastErrorKind="analysis_failed"`) instead of dropping it on
  the floor when the job itself succeeded.
- `classifyInitialReadError()` in `src/ingest/initial-read-status.ts`
  separates `backend_unavailable / route_error / version_mismatch /
  lag_tolerated / ok` so callers can record severity rather than a
  generic failure.
- A five-case shared negative-fixture set (`retry-scheduled-crawl /
  expired-running-lease / missing-analysis-after-crawl-success /
  failed-analysis-after-crawl-success / terminal-dead-crawl`) locks
  both layers offline with exact case-name set-equality enforced in
  CI (`tests/backend-job-status-projection.test.ts` +
  `tests/fixtures/backend-job-status-negative-cases.json`).
- `ProcessingStrip` and `TopicProcessingStatus` swap labels when
  recovery copy applies; the topic detail panel exposes a "重啟處理"
  affordance when `BackendWorkUiState.kind === "expired_running"`.

### Backend health dot

- `entrypoints/background.ts` `backend/get-health` handler now calls
  the lightweight backend `/health` endpoint (`fetchBackendHealth`,
  3 s timeout) instead of the slower `/worker/status`. The richer
  worker-status projection still drives backlog / retry / analysis
  copy on click-open.
- `useProcessingCoordinator` runs a `/health` heartbeat alongside the
  worker poll. Streak rule: 1 failure stays green (suppressed),
  2 consecutive = yellow, 3+ = red, any success returns to green.
- `ProcessingStrip` leading edge renders the dot with that streak
  state. Existing `BackendWorkUiState` projection rendering is
  unchanged.

### Threads extraction audit — extension half

- `src/targeting/threads.ts` descriptor gained
  `engagement.source: "card" | "page_fallback" | "missing"` as an
  additive metadata field; existing engagement numbers and selector
  logic are byte-identical.
- Descriptor fixture replay test
  (`tests/targeting-descriptor-fixtures.test.ts`) locks
  `findCardCandidate` / `buildTargetDescriptor` output for the rich
  thread fixture captured in the backend repo.
- Phase C deliverables landed: cross-layer extraction report and the
  ranked risk register
  (`docs/handoff/2026-06-16-threads-extraction-crawler-risk-register.md`).

### Docs

- Architecture map updated: `API` / `JOBS` moved from 🟡 to 🟢 (built,
  not locked); status preamble updated; class assignments updated.
- Memory docs (`docs/memory/current-state.md`,
  `docs/memory/latest-shared-context.md`) gained the earned-status
  paragraph for API/JOBS.
- New handoff docs: API/JOBS implementation plan, threads extraction
  audit + implementation plan, popup health-dot design note,
  threads cross-layer report, threads risk register.

## What shipped — backend (`dlens-ingest-core`)

### `/worker/status` work-truth summary

- `WorkerStatusResponse` extended with the same 12 additive fields
  the extension projects. `DatabaseWorkerControlService.get_status()`
  now joins liveness + a safe DB-read summary + the in-memory
  `_last_drain_error` / `_last_drain_finished_at` it captures from
  the drain thread.
- New `summarize_work_state()` on `PostgresJobStore` and
  `InMemoryJobStore`. Two aggregate SQL queries, both backed by
  existing `crawl_jobs` and `capture_analyses` indexes.
- Five backend negative fixtures matching the extension set.

### `/health` endpoint

- `src/dlens_ingest_core/api/routes/health.py` — new router. Returns
  `{status:"ok", uptime_seconds, process_id}`. No DB, no Playwright,
  no service injection. p99 < 20 ms in-process; ~500 ms cold round
  trip from the extension. The split with `/worker/status` (DB-bound,
  1–6 s on Supabase WAN) lets the health dot use a 3 s timeout
  without false-flagging slow DB connects.

### LaunchAgent autostart

- `scripts/launchd/com.dlens.ingest.plist.template` +
  `install.sh / uninstall.sh / status.sh / README.md`. Renders the
  plist with absolute paths into `~/Library/LaunchAgents/`, bootouts
  any existing instance, then bootstraps under `gui/<uid>`.
  `RunAtLoad=true`, `KeepAlive` on crash only (not on success),
  `ThrottleInterval=10`.
- The agent runs `python scripts/run_api.py` from the repo root, so
  `.env` flows through `load_dotenv()`. No secrets enter the plist.

### Threads extraction audit — backend half

- Phase A inventory landed under `docs/audit/`: static call graph,
  env-var ledger, artifacts map, browser action inventory,
  legacy-candidate ledger.
- Phase B fixture capture: one owner-authorized live crawl against
  `@aiposthub/post/DZpehhuAdCe`, sanitized and committed under
  `tests/crawlers/fixtures/threads/rich-thread/`. The capture script
  itself does not modify `run_fetcher_test()`; it skips drill to
  keep the action budget minimal.
- Action-budget manifest: `run_fetcher_test` now emits an
  `action_budget` block (pages opened, wheel events, expand clicks,
  drill stats, elapsed seconds, stop reason, env snapshot) inside
  `manifest.json`. Observation-only; loop bounds and defaults
  unchanged.
- Profile-hover policy test: `run_fetcher_test` cannot opt into
  `include_profile_metrics=True`. Enforced by a static check that
  would fire if a future refactor wires the hover path into the
  default crawl.
- Parser fixture replay test against the rich-thread fixture, locking
  current `vendor/parser.py:extract_data_from_html` output for the
  captured labels.
- Capture-hint mismatch telemetry: the adapter no longer `del`s
  `capture_hints`; it now reads them and emits non-blocking
  `capture_hint_mismatches` diagnostics for author / post-url-canonical
  / body-prefix / target-type. Mismatch never alters crawl outcome.

## Operational changes

- macOS users can install the backend LaunchAgent with
  `bash scripts/launchd/install.sh` from the backend repo; the
  extension finds the backend at `127.0.0.1:8000` without manual
  terminal work after that.
- Backend `/health` is the recommended liveness check; reserve
  `/worker/status` for actual work-state polling.

## Known gaps (deliberate)

- `CRAWLER` remains 🟡. The Phase C risk register at
  `docs/handoff/2026-06-16-threads-extraction-crawler-risk-register.md`
  is the source of truth for what's left. Three owner-approved live
  fixtures (`direct-reply`, `repost`, `quoted-post`) gate any further
  selector / parser change. The audit is explicit: changing selectors
  before those fixtures land is guessing.
- `SEAM_PARTIAL` and `CS` (content-script DOM extraction) remain 🟡 for
  the same reason `CRAWLER` does — they ride on Threads DOM behavior
  the local test surface cannot exercise.
- Profile-hover code path (`fetch_advanced_metrics →
  _extract_author_profile_metrics`) is reachable only through the
  `/threads/advanced-metrics` API endpoint, never from the default
  crawl. The policy test locks this boundary.
