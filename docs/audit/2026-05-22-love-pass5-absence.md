### B.1 最強結論：中間層缺席是 platform affordance，不是議題特性 [強]

work P6 §6/§7 把「消失的中間層」當成 work 的核心發現，並**保留了 alternative explanation：可能是 Threads platform affordance**。love 拆解了它：

**即使是 love 這種有真辯論、有陣營、有 ideology-critique 的 value-laden topic，集體 / 制度層仍然完全缺席。** love 的 reader 會反駁、會校正 OP 的價值觀、會指認「受資本影響」這種結構性語言——但**從不升級到「我們 / 社會 / 集體應該如何」**。退場（del app）和個人策略（回到現實空間）是唯一回應，沒有人要求修平台、改文化、組織起來。

→ **work 的「中間層缺席」的 alternative explanation 得到強烈支持**（2 個 topic 證據都指向平台層，但 2 個樣本不等於「證實」）：這較可能不是「哀悼型議題」特性，而是 **Threads 這個 platform 把 discourse 原子化的結構特徵**（短帖 + 演算法個人化推送 + 匿名 + 留言不形成 community）。兩個性質完全不同的 topic 都呈現同一個 individual ↔ structure 直接對撞、中間層真空。**仍不能排除更上層的文化因素**（非平台獨有），需更多 topic / 跨平台才能再收窄。

### B.2 第二強結論：work §6/§7 的其餘核心發現是 OVERFIT [強]

work 的編輯閱讀（P6 §7）命題是：「一個已經學會精準命名困境、卻系統性缺乏把困境轉成主張的語言、發言人和場域的 discourse；沒有 future tense；樂觀被當場撤銷信用。」

love 證明這組命題**大部分是 work 哀悼性格的 overfit，不是 Threads 通則**：
- love **有** future tense（與子偕老）
- love **有** prescription（靚女應該）
- love 的正面敘事**被擁抱**不被撤銷信用（S6）
- love **有**真辯論

→ 若直接拿 work 的 trace 寫 prompt，prompt 會把「無 future / 無 prescription / 壓制樂觀 / 不辯論」當成 audit 的通用發現——**這會是嚴重 overfit**。這四項必須是「每個 topic 重新發現」的，不能預設。

### B.3 跨 topic 共享但成因待定：量化 cognitive mode [中]

算帳 / 量化在兩個 topic 都是 default（work 算 17k / 裁員百分比，love 評分 / 收入編號男友）。這**可能**是更深的 HK 文化模式（把一切量化），也**可能**只是兩個 topic 都觸及經濟焦慮的巧合。
**[中] 理由：** 2 個 sample 不足以區分「文化模式」vs「兩個都碰錢的巧合」。需要一個不碰經濟的 topic（例如純興趣 / 寵物 / 飲食）才能定。**這是是否要做第 3 個 topic 的主要理由。**

## C. 對 prompt / schema 的直接影響

### C.1 prompt 必須區分「可預期的結構發現」vs「每 topic 重新發現的內容發現」
- **可作為 audit 預設去 probe 的（platform-level，跨 topic 穩）：** 中間層 / 集體 scale 是否缺席（B.1）。prompt 可以指示「檢查 discourse 是否停在 individual ↔ structure 兩極而缺中間」——這個 probe 跨 topic 有效。
- **必須每 topic 重新發現、不可預設的（topic-contingent）：** future tense 有無、prescription 方向、正面敘事的命運、是否有辯論（B.2）。prompt **不可**把 work 的「無 future / 壓制樂觀」寫成通用判讀指令。

### C.2 absence pass 的 prompt 要強制「測試而非假設」
work 的 absence 結論若沒有 love 對照，會被誤當通則。→ 單 topic audit 的 absence pass **必須標明哪些缺席是 sample-level、哪些可能是 platform-level，且不可宣稱 platform-level 除非有跨 topic 證據**。這正是 work P5 evidence-strength 分級的價值——它讓 work §6 的核心結論保留了 alternative explanation，使 love 能夠拆解它。

### C.3 cross-topic 是 audit 框架可移植性的必要驗證
沒有 love，work §6/§7 看起來是深刻洞察；有 love，才知道其中 4 項是 overfit、1 項是真 platform 特徵。→ **prompt 產品化後，「跨 topic 對照」應該是一個獨立功能 / 階段**，不能只靠單 topic audit 自我宣稱洞察。
